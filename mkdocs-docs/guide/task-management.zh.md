# 任务管理

OneManCompany 使用分层任务系统，具备依赖追踪、质量关卡和各层级的 CEO 审批机制。

## 任务类型

| 类型 | 描述 | 审批方式 |
| --- | --- | --- |
| **Simple** | 单步任务 | 完成后自动验收 |
| **Project** | 多步骤、含子任务 | 需要手动验收 + 复盘 |

## 任务生命周期

每个任务遵循统一的状态流转：

```
pending → processing ⇄ holding → completed → accepted → finished
```

- **pending** — 任务已创建，等待被领取
- **processing** — 员工正在执行中
- **holding** — 暂停中，等待输入或依赖
- **completed** — 工作已完成，等待评审
- **accepted** — CEO/管理者已批准产出
- **finished** — 完全关闭（Project 类型需完成复盘后关闭）

异常状态：

- **failed** — 可重试，可回到 processing
- **blocked** — 依赖项失败
- **cancelled** — 已终止

## 创建任务

作为 CEO，在控制台中输入你的需求：

> "设计一个新产品的落地页"

EA 会将其路由给合适的人。对于复杂需求，COO 会将其拆解为带依赖关系的任务树。

## 任务树

Project 会被自动拆解为子任务层级结构：

```
制作一个益智游戏 (Project)
├── 设计游戏机制 (Simple)
├── 创建美术素材 (Simple)
│   └── 依赖：设计游戏机制
├── 实现游戏逻辑 (Simple)
│   └── 依赖：设计游戏机制
├── QA 测试 (Simple)
│   └── 依赖：实现游戏逻辑、创建美术素材
└── 打磨并发布 (Simple)
    └── 依赖：QA 测试
```

依赖关系自动追踪 — 当依赖项达到 `accepted` 或 `finished` 状态时，子任务会自动解除阻塞。

## 评审工作成果

当员工完成任务时：

1. 你会在 CEO 控制台收到通知
2. 查看交付物和文件变更
3. **Accept（接受）** 以批准，或 **Reject（驳回）** 并附上修改意见
4. 被驳回的任务会退回给员工进行迭代

## 质量关卡

每个层级都有质量关卡：

- **员工** → COO 评审子任务产出
- **COO** → EA 评审项目级交付物
- **EA** → CEO 进行最终审批

## 复盘

Project 类型的任务在验收后会自动触发复盘：

- 做得好的地方
- 可以改进的地方
- 经验教训

这些洞察会被提炼到员工的工作原则和公司知识库中，推动持续改进。
