# 文件变更审批

当 AI 员工修改文件时，变更会经过评审和审批流程 — 就像真实工程团队中的代码评审一样。

## 工作原理

1. **员工提交变更** — 在任务执行过程中，员工可能创建或修改文件
2. **变更暂存** — 文件修改被排队等待 CEO 评审，不会立即生效
3. **CEO 评审 diff** — 你可以逐行查看具体改动
4. **批量审批** — 可以逐个或批量接受/拒绝变更

## 你会看到什么

对于每个待审变更：

- **文件路径** — 哪个文件被修改
- **Diff 视图** — 新增行（绿色）、删除行（红色）、上下文
- **员工** — 谁做了变更以及原因
- **任务上下文** — 哪个任务触发了变更

## 审批操作

| 操作 | 效果 |
| --- | --- |
| **Accept（接受）** | 变更被应用到文件 |
| **Reject（拒绝）** | 变更被丢弃，通知员工 |
| **Accept all（全部接受）** | 批量批准所有待审变更 |

## 为什么这很重要

文件审批是关键的质量关卡：

- **发现错误** — AI 员工可能犯错；评审可以防止错误上线
- **保持一致性** — 确保变更符合你的代码库标准
- **保持知情** — 作为 CEO，你始终掌握项目中的变更动态
- **安全网** — 不会有未经授权的重要文件修改

## 建议

!!! tip "定期评审"
    不要让待审批的变更堆积 — 它们可能阻塞员工在依赖任务上的进度。

!!! tip "信任但验证"
    随着员工通过辅导和绩效评估证明了可靠性，你可以选择降低评审频率。但安全网始终在那里。
