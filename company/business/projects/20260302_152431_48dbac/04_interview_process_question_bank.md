# 面试流程设计（含题库）— 代码工程师（Engineer）

## 0) 版本与使用说明
- 当前版本：v0.2
- 目标：将“流程 + 面试官分工 + 题纲 + 评分口径 + SLA”沉淀为唯一执行依据。
- 变更：若 CEO 确认路线/技术栈变化，请先更新 `02_JD_compensation.md` 的「路线与技术栈」后，再同步微调本题库。

---

## 总体原则
- 总时长控制在 **1.5–3 天** 完成（适应狼性文化与小团队快节奏）。
- 以“**可上线的交付能力**”为主线：工程化、质量、稳定性、可迭代性。
- 面试结论必须可追溯：每轮有评分表、证据（作业仓库/代码片段/面试记录）。

---

## 1) 流程概览（推荐）
1. **简历/作品集筛选（HR 15min）**
2. **电话/视频初筛（30min）**：技术栈核对 + 项目深挖
3. **作业/笔试（2–3h，带时间盒）**：后端 + LLM/Agent 小实现
4. **技术面（60–90min）**：代码走查 + 系统设计 + Debug
5. **文化/价值观面（30–45min）**：ownership、狼性文化、抗压与沟通
6. **CEO 终面（30min）**：目标对齐、薪酬、入职时间

> 备选：若候选人资深且作品突出，可跳过作业，改为“现场 pairing + 代码走查”。

---

## 2) 面试官安排与分工（内部执行）
> 目的：确保每轮“谁来问什么”固定，避免主观漂移；若排期冲突，允许替补但需沿用同一评分表。

| 轮次 | 时长 | 面试官/角色 | 产出物 | 通过门槛（建议） |
|---|---:|---|---|---|
| HR 初筛 | 15–30m | Sam HR（00002） | 初筛记录 + 结论 | Must-have 满足度 ≥ 80% |
| 技术初筛 | 30m | Alex COO（00003） | 技术初筛记录 | 有真实上线/交付证据 + 能讲清边界 |
| 作业/笔试 | 2–3h | 候选人（HR 发题与收件） | 作业 repo/压缩包 + 运行说明 | 可运行 + 结构清晰 + 有测试 |
| 技术面 | 60–90m | Alex COO（00003）主面；HR旁听记录 | 评分表 + 关键证据链接 | 综合 ≥ 3/5 且无淘汰项 |
| 文化面 | 30–45m | Sam HR（00002） | 文化面记录 | Ownership/抗压/沟通无明显风险 |
| CEO 终面 | 30m | CEO | 终面结论 | 目标/薪酬/入职时间达成一致 |

### 面试反馈 SLA
- 每轮结束后 **48 小时内** 必须输出结论与评分（通过/保留/拒绝）并回传 HR。

---

## 3) 统一评分维度（建议）
每个维度 1–5 分，写清证据；出现“一票否决”直接拒绝。

1. **独立交付与上线闭环**（需求澄清→开发→测试→部署→监控/复盘）
2. **工程化与质量意识**（测试、CI、日志、可维护性、代码风格）
3. **后端基础能力**（API 设计、DB、异步、性能与可靠性）
4. **LLM/Agent 落地能力**（结构化 prompt、工具调用、评测与回归）
5. **沟通与 Ownership**（问题闭环、透明度、能写文档）

### 一票否决（淘汰信号）
- 只会“写片段”无法解释上线/稳定性保障
- 无测试/质量意识且不认为是问题
- 无法说明个人贡献边界，只有“团队做了什么”
- 对 LLM 仅停留在“会用聊天”，无结构化/可评测迭代思路

---

## 4) 初筛（30min）问题清单
- 请用 3 分钟讲一个你主导上线的项目：你负责的边界是什么？
- 项目里最难的 bug/故障是什么？你如何定位与复盘？
- 你对测试/CI 的理解：你会优先补哪些测试？
- 你做过的 LLM 功能：如何评估效果（离线/在线）？如何回归？
- 你在不确定需求下如何推进？如何与业务方对齐？

---

## 5) 作业/笔试（2–3h）
### 作业目标
实现一个最小可用的“Agent 工具调用 API 服务”，强调工程化。

### 题目（版本 A：偏后端 + 工程化）
- 用 FastAPI 实现：
  - `POST /tasks` 创建任务（入参：title, content）
  - `GET /tasks/{id}` 查询任务
  - `POST /agent/run`：输入用户问题，服务端调用一个“工具函数”（mock）返回结果
- 要求：
  - Pydantic 校验、清晰的项目结构
  - 至少 3 个单元测试（pytest）
  - Dockerfile 或可运行说明
  - 基础日志（结构化更佳）

### 题目（版本 B：偏 LLM/RAG）
- 设计一个简单 RAG：
  - 支持上传 3–5 段文本
  - embedding + 向量检索（可用任意轻量方案/本地 mock）
  - `POST /ask` 返回：检索到的片段 + 生成答案（LLM 可 mock）
- 要求：
  - 给出评测思路（至少 3 条用例）

### 评分 Rubric（建议）
- 工程结构/可运行性 30%
- API 设计与健壮性 20%
- 测试与可维护性 25%
- LLM/Agent 思路（结构化、可评测）25%

---

## 6) 技术面（60–90min）题库
### (1) 代码走查
- 让候选人讲解作业结构：为何这样分层？
- 追问：如何处理异常、超时、重试、幂等？

### (2) 系统设计（任选其一）
1. **Agent 平台最小架构**：
   - 如何组织 prompt、工具、记忆、RAG？
   - 如何做评测与回归？
2. **异步任务系统**：
   - API 接收请求 → 投递队列 → worker 执行 → 回调/轮询
   - 如何保证可观测性与失败重试？

### (3) Debug/排障
- 给一个 FastAPI 接口 500/超时的案例（口述或伪日志）：
  - 候选人说出定位路径：日志 → 复现 → 指标 → 二分 → 修复 → 回归

### (4) 基础知识点（抽问）
- Python async：何时用协程？常见坑（阻塞 IO、线程池）
- 数据库：索引为何能加速？如何避免 N+1？
- API：分页/鉴权/版本化
- Docker：镜像分层、如何减小镜像体积

---

## 7) 文化/价值观面（30–45min）题库
- 你经历过最强压力的一次交付，如何确保结果？
- 如果 CEO 的需求模糊且经常变化，你如何推进并管理预期？
- 你如何看待“速度 vs 质量”的取舍？给真实案例。
- 你最在意的工作方式/原则是什么？哪些情况你会拒绝？

---

## 8) 背调建议（可选）
- 核验：实际负责范围、上线经验、故障处理、与同事协作口碑。
