# 面试流程设计（含题库）— 代码工程师（Engineer）

## 总体原则
- 总时长控制在 **1.5–3 天** 完成（适应狼性文化与小团队快节奏）。
- 以“**可上线的交付能力**”为主线：工程化、质量、稳定性、可迭代性。

---

## 流程概览（推荐）
1. **简历/作品集筛选（HR 15min）**
2. **电话/视频初筛（30min）**：技术栈核对 + 项目深挖
3. **作业/笔试（2–3h，带时间盒）**：后端 + LLM/Agent 小实现
4. **技术面（60–90min）**：代码走查 + 系统设计 + Debug
5. **文化/价值观面（30–45min）**：ownership、狼性文化、抗压与沟通
6. **CEO 终面（30min）**：目标对齐、薪酬、入职时间

> 备选：若候选人资深且作品突出，可跳过作业，改为“现场 pairing + 代码走查”。

---

## 1) 初筛（30min）问题清单
- 请用 3 分钟讲一个你主导上线的项目：你负责的边界是什么？
- 项目里最难的 bug/故障是什么？你如何定位与复盘？
- 你对测试/CI 的理解：你会优先补哪些测试？
- 你做过的 LLM 功能：如何评估效果（离线/在线）？如何回归？
- 你在不确定需求下如何推进？如何与业务方对齐？

---

## 2) 作业/笔试（2–3h）
### 作业目标
实现一个最小可用的“Agent 工具调用 API 服务”，强调工程化。

### 题目（版本 A：偏后端 + 工程化）
- 用 FastAPI 实现：
  - `POST /tasks` 创建任务（入参：title, content）
  - `GET /tasks/{id}` 查询任务
  - `POST /agent/run`：输入用户问题，服务端调用一个“工具函数”（mock）返回结果
- 要求：
  - Pydantic 校验、清晰的项目结构
  - 至少 3 个单元测试（pytest）
  - Dockerfile 或可运行说明
  - 基础日志（结构化更佳）

### 题目（版本 B：偏 LLM/RAG）
- 设计一个简单 RAG：
  - 支持上传 3–5 段文本
  - embedding + 向量检索（可用任意轻量方案/本地 mock）
  - `POST /ask` 返回：检索到的片段 + 生成答案（LLM 可 mock）
- 要求：
  - 给出评测思路（至少 3 条用例）

### 评分 Rubric（建议）
- 工程结构/可运行性 30%
- API 设计与健壮性 20%
- 测试与可维护性 25%
- LLM/Agent 思路（结构化、可评测）25%

---

## 3) 技术面（60–90min）题库
### (1) 代码走查
- 让候选人讲解作业结构：为何这样分层？
- 追问：如何处理异常、超时、重试、幂等？

### (2) 系统设计（任选其一）
1. **Agent 平台最小架构**：
   - 如何组织 prompt、工具、记忆、RAG？
   - 如何做评测与回归？
2. **异步任务系统**：
   - API 接收请求 → 投递队列 → worker 执行 → 回调/轮询
   - 如何保证可观测性与失败重试？

### (3) Debug/排障
- 给一个 FastAPI 接口 500/超时的案例（口述或伪日志）：
  - 候选人说出定位路径：日志 → 复现 → 指标 → 二分 → 修复 → 回归

### (4) 基础知识点（抽问）
- Python async：何时用协程？常见坑（阻塞 IO、线程池）
- 数据库：索引为何能加速？如何避免 N+1？
- API：分页/鉴权/版本化
- Docker：镜像分层、如何减小镜像体积

---

## 4) 文化/价值观面（30–45min）题库
- 你经历过最强压力的一次交付，如何确保结果？
- 如果 CEO 的需求模糊且经常变化，你如何推进并管理预期？
- 你如何看待“速度 vs 质量”的取舍？给真实案例。
- 你最在意的工作方式/原则是什么？哪些情况你会拒绝？

淘汰信号：
- 推卸责任、只描述“团队做了什么”而无个人贡献证据
- 对质量无感（不写测试、不做复盘）
- 过度追求完美导致无法交付

---

## 5) 背调建议（可选）
- 核验：实际负责范围、上线经验、故障处理、与同事协作口碑。
