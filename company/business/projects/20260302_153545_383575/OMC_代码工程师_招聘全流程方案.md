# One Man Company — 代码工程师招聘全流程方案（HR）

> 目的：在 1–2 周内启动并稳定运转“需求澄清 → JD → 渠道投放 → 筛选/面试 → 漏斗/周报 → offer”的闭环，尽快拿到可面试的有效候选人，并在 2–4 周内完成入职。

---

## 1) 与 CEO / 用人方澄清岗位画像（问题清单 + 当前默认假设）

### 1.1 必须向 CEO 确认的 7 项关键信息（请 CEO 直接逐条回复）
1. **岗位定位**：后端为主 / 全栈 / 基础设施？（主战场是什么）
2. **核心交付物（未来 30/60/90 天）**：
   - 30 天：___（例：完成服务骨架、数据库建模、CI、核心 API v1 上线）
   - 60 天：___（例：关键业务流程闭环、可观测性、性能优化）
   - 90 天：___（例：稳定迭代节奏、技术债清理、关键指标达标）
3. **技术栈硬约束**：
   - 必须：Python？FastAPI/Flask？Postgres？Docker？
   - 可替代：Node/Go 是否接受？ORM/框架可替代范围？
4. **级别**：优先招 **初级/中级/高级** 哪个档？（我们默认：中级偏上更稳）
5. **地点/远程**：
   - 远程是否 100% 可？是否需要来访？
   - **时区重叠**：每天与中国时区重叠至少 __ 小时（建议 3–4h）
6. **薪资范围 + 期权**：
   - Base（月薪/年薪）：__
   - 奖金：__
   - 期权/虚拟股：__
7. **试用期与入职时间**：
   - 试用期：__（建议 3 个月）
   - 最晚入职时间：__（建议 4 周内）

### 1.2 当前已对齐的执行原则（与 COO/EA 的澄清会结论）
- 画像必须落到可执行链路：**岗位目标/交付物 → 能力要求 → 级别薪资 → 面试流程产能**。
- 提前锁定硬约束：**必须技术栈、远程时区重叠、最晚入职时间、30/60/90 验收指标**。
- 面试执行参数先定：**轮次、每周面试位、反馈 SLA（24–48h）、作业题耗时上限（2–3h）**。

### 1.3 默认假设（若 CEO 未明确，先按以下版本启动）
- **岗位**：后端/全栈偏后端代码工程师（能独立交付后端服务与内部工具）
- **远程**：远程优先，**需与中国时区重叠 ≥3–4h/天**
- **入职**：2–4 周内
- **试用期**：3 个月（按 30/60/90 天验收）
- **级别**：偏中级（Lv.2 能力），能独立推进需求到上线

## 4) 筛选标准与面试流程

### 4.1 简历筛选要点（必须项/淘汰项）
**必须项（任一缺失都显著降级）**
- Python 后端项目（对外 API 或内部服务均可），说明个人贡献
- SQL 能力：表设计 / 索引 / 性能意识（至少能说清优化思路）
- 工程化习惯：测试、lint、代码规范、日志
- 远程协作证据：异步沟通、文档、issue/PR 流程

**强淘汰项（Red flags）**
- 项目经历“只写业务代码”，对上线/运维/故障处理完全陌生且无学习证据
- 没有任何可验证产出（代码仓库/项目描述含糊、无法讲清难点与结果）
- 频繁跳槽且解释不清，或远程协作不适配（响应慢、不可被管理）

### 4.2 面试流程（建议 3–4 轮，控制周期 ≤7 天）
**Round 0：HR 预筛（15–20min）**
- 动机/稳定性/远程条件/到岗时间/薪资期望

**Round 1：技术面（60min，工程能力为主）**
- Python 基础与工程实践、API 设计、数据库建模/SQL、错误处理与可观测性

**Round 2：作业题（Take-home，2–3h，上限控制）**
- 交付一个可运行的小服务（FastAPI）+ 数据库 schema + 2–3 个接口 + 单测 + README
- 评分重点：可运行、可读性、测试、边界条件、日志、工程结构

**Round 3：综合面/终面（30–45min，CEO/用人方）**
- 价值观/ownership/沟通方式/过去最难问题复盘/30-60-90 plan

> 反馈 SLA：每轮面试后 **24–48h 内**给出结论；避免候选人流失。

### 4.3 作业题建议（示例）
题目：实现一个“任务队列/待办”服务（简化版）
- API：创建任务、查询任务列表（分页/过滤）、更新状态
- 数据库：Tasks 表（含状态/创建时间/更新时间/幂等键）
- 要求：
  - FastAPI + SQLAlchemy 或纯 SQL
  - 单元测试 3 个以上
  - Dockerfile + README（如何启动、如何跑测试）
  - 日志（至少请求/错误日志）

可选：对高年资候选人增加“简单系统设计问答”替代加班式作业题。

### 4.4 评分表（Scorecard，面试官统一口径）
总分 100（建议通过线 75）
- A. 工程实现能力（30）：代码结构、可维护性、抽象与边界
- B. API 设计与后端基础（20）：REST、鉴权思路、错误码、幂等
- C. 数据库与 SQL（20）：建模、索引、查询优化意识
- D. 质量与可靠性（15）：测试、日志、监控、异常处理
- E. 沟通与 ownership（15）：问题澄清、推进能力、远程协作

面试结论：Strong Yes / Yes / Hold / No（同时给出风险点与补救建议）

## 5) 候选人漏斗与周报机制

### 5.1 漏斗定义（每周更新）
- 触达（曝光/主动搜索）
- 投递/简历入库
- 简历合格（进入面试）
- HR 预筛通过
- 技术面通过
- 作业题通过
- 终面通过
- Offer 发出
- Offer 接受
- 入职

### 5.2 指标与目标（初版，可按实际数据滚动调整）
- 简历合格率（合格/投递）：目标 15–25%
- 技术面通过率（通过/技术面）：目标 30–50%
- Offer 接受率：目标 60–80%
- 周面试产能：技术面 3–5 人/周（小团队建议先保守）

### 5.3 周报机制（模板）
**每周一 10:00（或 CEO 指定）邮件/飞书同步：**
- 本周漏斗数据（本周新增/累计）
- 在途候选人列表（下一步、面试时间、阻塞点）
- 本周问题与风险（画像是否偏、薪资是否卡、流程是否慢）
- 下周计划（渠道、面试排期、需要 CEO 决策项）

## 6) 1–2 周内可落地的行动计划 + 预算

### 6.1 1 周启动计划（Day 1–7）
- **Day 1**（今天）：
  - CEO 回复“画像 7 问”并确认最终 JD（或确认默认假设）
  - 定面试官与排班；确定反馈 SLA 与轮次
- **Day 2**：
  - 发布渠道上线（BOSS/拉勾/社区）；内推文案发出
  - 建候选人 tracking 表（含漏斗阶段、负责人、下一步时间）
- **Day 3–5**：
  - 每天筛简历 + 安排 HR 预筛/技术面（保持滚动）
  - 固定每日 30min 招聘站会（HR + COO/用人方）
- **Day 6–7**：
  - 第一批候选人发作业题；收集结果并校准评分标准

### 6.2 第 2 周（Day 8–14）
- 保持每周 3–5 场技术面；对通过者在 7 天内完成终面
- 针对数据调整：
  - 合格率过低：调整 JD/渠道/关键词
  - 通过率过低：校准题目难度与画像
  - 接受率低：校准薪资带宽与卖点

### 6.3 预算建议（2 周内）
> 以“先跑通闭环、再加预算”为原则。
- BOSS 直聘：¥3,000–6,000（按套餐）
- 拉勾（可选）：¥2,000–4,000
- 内推奖励：¥2,000/人（通过试用期后发放；2 周内不一定产生现金流出）
- 作业题补贴（可选，提升体验，针对资深）：¥200–500/人 * 5 人 = ¥1,000–2,500

**两周启动预算合计（建议上限）：¥6,000–12,000**（不含内推奖励的后置支出）
