# Take-home 作业说明与提交规范

## 1. 背景与目标
感谢您应聘 One Man Company 软件工程师岗位！为了更好地了解您的实际编码能力、工程化习惯以及独立交付闭环的能力，我们为您准备了一份小型的 Take-home 作业。
本作业预计耗时 **2-4 小时**，请在收到本说明后的 **48 小时内**完成并提交。

## 2. 任务要求
请实现一个**小型 API 服务**。
- **技术栈要求**：Python (FastAPI) 或 Node.js (Express/NestJS)。
- **功能要求**：实现一个简单的资源增删改查 (CRUD) 接口（具体业务对象可自定义，如“任务列表”、“用户管理”等）。
- **工程化要求**：
  - 包含输入数据校验。
  - 包含至少 2-3 个核心单元测试 (pytest 或 jest)。
  - 提供 Dockerfile，使得该服务可以通过 Docker 容器运行。
  - 编写 README.md 文件。

## 3. README.md 规范 (必须包含)
提交的仓库中必须包含一份清晰的 README.md，内容应涵盖：
1. **项目简介**：说明这是什么服务，提供了哪些 API。
2. **启动说明**：如何本地运行代码（含依赖安装及环境变量配置说明）。
3. **Docker 运行说明**：如何通过 Docker 构建和运行。
4. **测试说明**：如何运行单元测试。
5. **设计思路/Trade-off (简述)**：在实现过程中您做出的关键技术决策或妥协（例如为何选择某种数据结构，或者如果时间充足会做哪些优化）。

## 4. 提交规范
- 请将代码提交至一个 **公开的 GitHub 仓库**（或提供私有仓库访问权限）。
- 将仓库链接通过邮件/消息回复给 HR。
- 提交前请确保按照 README 中的启动说明，可以在纯净环境中顺利跑通。

## 5. 评分要点 (我们关注什么)
- **正确性**：API 功能是否符合预期，是否处理了基本的边界情况。
- **可读性与规范性**：代码结构是否清晰，命名是否规范。
- **工程化意识**：测试覆盖情况，Docker 容器化是否可用，README 是否完善、对协作者友好。
- **防作弊机制**：在后续的复试中，面试官会抽取 15 分钟与您进行 Pairing/追问关键实现细节，请确保您对提交的代码完全理解。
