# 面试流程、用人优先级与预算区间（COO建议稿）

## 1) 用人优先级（结论先行）
**优先级 P0：后端偏重的全栈（Backend-leaning Full-stack）**
- 原因：小团队最缺的是“能把一个功能从 API+DB→上线→排障”闭环的人；纯前端/纯 DevOps 在现阶段 ROI 较低（除非已有明确瓶颈）。

**P1：纯后端（如果产品前端相对简单/可由现有人员覆盖）**
- 适配：API、数据建模、性能稳定性、异步任务/队列等。

**P2：DevOps/平台（当出现明确交付阻塞时再招）**
- 触发条件：发布频率高但事故多；环境/CI 成本过高；监控缺失导致排障慢。

**P3：前端（当有明确复杂 UI/增长投放/运营后台需求时再招）**

> 若 CEO 明确未来 3–6 个月的最大风险在“交付速度与线上稳定性”，P0 不变；若最大风险在“增长/交互体验”，再考虑 P3 上升。

## 2) 预算区间（建议“能招到”为目标，避免口径虚）
> 与 `OMC_Engineer_Hiring_Package.md` 对齐，并给出可执行边界。

### 国内全职（建议）
- **目标区间**：25k–40k RMB/月（13–15薪折算）
- **上浮条件（到 45k–50k）**：
  - 同时具备：后端交付闭环 + 工程化/DevOps能力 + 能独立扛模块上线
- **下探条件（到 20k–25k）**：
  - 作为“试用型”候选人，先按项目/试用付费验证，转正再回到目标带宽

### 海外/远程（美元计价）
- **年薪**：$45k–$85k
- 或：$25–$55/h（先做 2–4 周付费试用项目，降低错配风险）

### 报价策略（运营口径）
- 对外可写“薪资面议/有竞争力”，但 HR 内部必须维护 **“上限”**，否则漏斗做不起来。

## 3) 面试流程（建议固定为 4 轮 + 可选背调）
> 原则：候选人从投递到终面 **≤10天**；每轮有“输入/输出/淘汰红线”。

### Stage 0：简历筛选 + 异步问卷（HR，10min）
- 输出：是否进入初筛；记录拒绝理由（技术栈不符/交付闭环不足/沟通风险等）。

### Stage 1：初筛（HR，30min）
- 目标：动机、经历核验、沟通、ownership、薪资与到岗。
- 淘汰红线：讲不清个人贡献；排斥测试/文档/上线责任。

### Stage 2：技术面-编码与基础（COO，60min）
- 结构：15min 快问快答（API/SQL/语言）+ 35min 现场编码 + 10min code review。
- 通过标准：能写出正确、可读、可测试的代码；边界条件意识。

### Stage 3：系统设计 & 工程化（COO +（可选）CEO，60min）
- 题型：服务设计、幂等/重试、发布回滚、日志与监控、故障排查。
- 通过标准：能权衡方案并讲清 trade-off；能提出可落地的工程化措施。

### Stage 4：终面（CEO，30–45min）
- 目标：价值观与抗压、目标对齐、决策速度与承诺。

### Stage 5（可选）：背调（HR，2人）
- 触发条件：候选人关键经历无法验证/风险较高/薪资在上沿。

## 4) 统一评分与决策阈值
- 建议 100 分制 + 强制结论：Strong Hire / Hire / Hold / No Hire
- **硬门槛**（任一不满足直接 No Hire）：
  - 不能独立完成编码题（或靠提示才能推进核心逻辑）
  - 对上线、排障、写测试有明显抵触
  - 沟通无法对齐事实与证据（说不清“做了什么→如何验证”）

## 5) 面试作业（48h take-home）建议
- 建议在 Stage2 通过后发放，控制在 2–4 小时工作量。
- 题目：实现一个小型 API 服务（FastAPI 或 Node），含校验、测试、Dockerfile、README。
- 防作弊：复试时做 15min pairing/追问关键实现细节。
