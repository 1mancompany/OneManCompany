# One Man Company（OMC）代码工程师招聘包（端到端）

> 版本：v1.0（2026-03-02）
> 
> 适用：CEO/HR/用人方统一执行；默认远程/线下均可（以岗位设定为准）。

---

## 1) 岗位画像（画像/职责/技能/级别/薪资建议）

### 1.1 岗位定位
- **角色（Role）**：Engineer（软件工程师 / 代码工程师）
- **部门（Department）**：技术研发部（Engineering）
- **职级（Level）**：**Lv.1 初级工程师**（公司制度：新入职统一 Lv.1）
- **用人目标**：在极小团队中，能端到端交付功能：从需求澄清 → 方案设计 → 编码 → 测试 → 部署/监控 → 文档沉淀。
- **工作方式**：高自驱、强 ownership；符合公司价值观“实事求是，求是创新；狼性文化，适者生存”。

### 1.2 核心职责（Responsibilities）
1. 负责核心业务功能的后端/服务端实现：API 设计、业务逻辑、数据模型。
2. 与产品/运营（或 CEO）直接对齐需求，拆解任务并按迭代交付。
3. 数据存储与查询：SQL 设计、索引、迁移脚本、性能优化。
4. 工程质量：单元/集成测试、代码 review、静态检查、文档。
5. Dev 基础：Docker 化、CI（GitHub Actions 等）、上线与回滚、日志与监控。
6. 故障排查与线上问题响应（以值班/按需为主）。

### 1.3 必备技能（Must-have）
- 熟练掌握 **Python 或 TypeScript/Node.js 至少一种**（最好两者都能读写）。
- 熟悉 **RESTful API** 设计与实现；理解鉴权、分页、幂等、错误码规范。
- 熟练 **SQL（PostgreSQL/MySQL）**：表设计、查询优化、事务。
- 熟练 **Git** 与常规协作流（PR、rebase、tag、release）。
- 有 **测试意识与实践**（pytest/jest，覆盖核心逻辑）。
- 能独立完成从 0→1 的小模块交付，沟通清晰，能写文档。

### 1.4 加分技能（Nice-to-have）
- Web 框架：FastAPI/Django/Flask 或 NestJS/Express。
- 基础设施：Docker、CI/CD、Linux、Nginx、基础可观测性（logs/metrics/traces）。
- 云服务经验：AWS/GCP/阿里云（任一）。
- LLM 应用开发：RAG、向量库、prompt/agent 工程、评测。
- 性能与稳定性：缓存（Redis）、队列（RabbitMQ/Kafka）、限流熔断。

### 1.5 软技能与行为标准
- 以结果为导向：能把“不确定”拆成可执行清单。
- 实事求是：遇到问题能给出证据（日志、复现、基准测试）。
- 狼性文化适配：抗压、能主动推进、遇到阻碍会升级沟通而非等待。

### 1.6 薪资范围建议（供 CEO 定价）
> 由于地区差异较大，以下按“市场中位偏上、但不溢价买资历”的策略。

- **国内一线/新一线（全职）**：
  - 月薪：**25k–40k RMB**（13–15 薪，视能力浮动）
  - 年包：**33–60w RMB**
- **海外/远程（美元计价）**：
  - 年薪：**$45k–$85k**（或按 $25–$55/h 外包试用转正）
- **建议策略**：
  - 允许“**试用期 1–2 个月按项目/按周付费**”降低风险；通过后转全职。
  - 若候选人具备 DevOps/全栈/LLM 复合能力，可在区间上沿。

## 3) 招聘渠道与节奏（内推/平台/社群等）

### 3.1 渠道组合（建议）
1. **内推（优先）**：创始团队朋友圈/前同事群/技术社群私域。
2. **平台**：Boss 直聘 / 拉勾 / 猎聘（国内）；LinkedIn / Wellfound（海外）。
3. **社群**：
   - Python/FastAPI/Node 技术群、开源社区（GitHub 项目 issues/discord）。
   - AI/LLM 工程社群（RAG、LangChain/LlamaIndex 相关）。
4. **开源定向挖掘**：直接筛选贡献者（commit 质量、issue 讨论质量）。

### 3.2 节奏（2 周冲刺版）
- **D0–D1**：发布 JD；准备题库/作业；搭建候选人漏斗表。
- **D2–D7**：简历筛选 + 电话/视频初筛（每天安排 3–5 个）。
- **D4–D10**：技术面（滚动进行）；同步发放作业（48h）。
- **D8–D12**：复试（系统设计/工程化）；必要时加 30min pairing。
- **D10–D14**：终面（CEO）+ 背调（如需）+ 发 offer。

KPI 建议：
- 每周触达 60–120 人（含被动投递+主动寻访）。
- 初筛通过率目标：20–30%；作业提交率：50–70%；offer 接受率：>60%。

## 4) 面试流程设计（筛选-技术面-复试-终面）+ 题库/作业/评分表

### 4.1 总流程（建议固定）
1. **简历筛选（HR/用人方 10min）**
2. **初筛（30min）**：动机/经历核实/沟通与 ownership
3. **技术面（60min）**：编码与基础能力
4. **复试（60min）**：系统设计 + 工程化（测试/部署/可观测）
5. **终面（30–45min，CEO）**：文化价值观/抗压/期望对齐
6. **背调（可选）**：1–2 位前同事/主管

> 原则：同一候选人从投递到终面不超过 10 天，避免流失。

### 4.2 初筛提纲（30min）
- 最近一个项目：你具体负责什么？交付周期？怎么验收？
- 遇到最棘手的 bug/事故：怎么定位？有什么证据链？
- 自驱：没有明确需求时你如何推进？如何和 CEO/PM 对齐？
- 期望：到岗时间、薪资范围、远程/通勤、成长诉求。

淘汰红线：
- 无法说明自己贡献；只讲团队不讲个人动作。
- 明显排斥写测试/文档/上线责任。

### 4.3 技术面（60min）题库建议
**A. 快速问答（15min）**
- REST API：幂等如何做？分页有哪些坑？错误码与异常如何设计？
- SQL：如何避免慢查询？索引何时失效？事务隔离级别理解。
- Python/TS：异步模型（async/await）理解；常见性能坑。

**B. 现场编码（35min）**（二选一）
- 题 1：实现一个带缓存与过期的 LRU / TTL cache（关注边界条件与测试）
- 题 2：实现一个简单的 API handler：校验输入 → 写入 DB（伪）→ 返回结构化错误

**C. 代码阅读（10min）**
- 给一段有 bug 的代码（并发/空指针/异常吞噬），让候选人指出问题与修复思路。

### 4.4 作业（Take-home，48 小时完成，2–4 小时工作量）
**作业题（推荐）**：
- 用 FastAPI（或 Node/NestJS）实现一个“任务管理”小服务：
  - 功能：创建任务、列表分页、更新状态、搜索（title contains）
  - 约束：
    - 输入校验（pydantic/zod）
    - 单元测试覆盖核心逻辑（至少 5 个用例）
    - Dockerfile + README（如何启动、如何跑测试）
    - 简单的错误处理与日志

**提交物**：Git 仓库链接或 zip。

**作业评分要点**：可运行性、代码结构、测试、边界处理、文档质量。

### 4.5 复试（系统设计/工程化，60min）
- 设计题：
  - “我们要做一个可扩展的 webhook 事件接收与重试系统”，你怎么设计？
  - 关键点：幂等键、重试策略、死信队列、监控指标、数据模型。
- 工程化追问：
  - CI 怎么做？发布怎么做？如何回滚？如何做分环境配置？
  - 你会记录哪些指标/日志字段来排障？

### 4.6 终面（CEO，30–45min）
- 价值观与行为：实事求是、面对不确定性推进能力。
- “狼性文化”适配：遇到阻碍时如何升级/协作；对高标准的态度。
- 目标与激励：想在 6 个月内做到什么？你最在乎什么？

### 4.7 统一评分表（建议 100 分制 + 强制结论）
| 维度 | 权重 | 评分标准（示例） |
|---|---:|---|
| 基础编码能力 | 25 | 代码正确性、可读性、复杂度控制 |
| 工程化/质量 | 20 | 测试、错误处理、日志、规范 |
| 数据与 API 能力 | 15 | SQL/数据建模、API 设计、边界意识 |
| 系统设计与可扩展性 | 15 | 能拆解问题，权衡方案 |
| Ownership/自驱 | 15 | 主动推进、复盘、对结果负责 |
| 沟通与协作 | 10 | 表达清晰、对齐能力 |

**结论（必须选一）**：Strong Hire / Hire / Hold / No Hire

## 6) 风险点与备选方案

### 6.1 常见风险点
1. **候选人“能写代码但不扛事”**：不愿意负责上线/监控/排障。
2. **简历水分**：项目描述宏大但贡献不清晰。
3. **面试通过但 offer 流失**：决策周期过长、薪资/远程政策不明确。
4. **作业作弊/代写**：代码风格与面试表现不一致。
5. **技术栈偏差**：只会某框架，基础薄弱，迁移能力差。

### 6.2 备选方案（建议同时推进）
- **方案 A：合同工/外包试用转正**：先用 2–4 周项目验证交付与沟通。
- **方案 B：招聘“全栈/偏后端”优先**：小团队更需要覆盖面。
- **方案 C：若短期招不到**：
  - 先补一位 DevOps/工程效率支持（但当前 open positions 优先级低）；或
  - 通过短期顾问完善 CI/部署，让工程师更聚焦业务。
- **方案 D：提高吸引力**：明确技术空间（LLM/数据）、更快决策、可远程。
